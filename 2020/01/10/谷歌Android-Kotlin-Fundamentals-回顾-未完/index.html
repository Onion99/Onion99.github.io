LifeCycle

在App复杂的情况，我们可能需要在Activity的onStart() 和 onStop() 回调中编写许多准备和释放操作代码, LifeCycle库正是用于对这种情况作出简化，LifeCycle库会监视所处的生命周期的变化，执行对应对应的操作

  • Lifecycle owners，即拥有生命周期的组件，比如Activity和Fragment
  • Lifecycle owners implement Lifecycle Owner接口
  • Lifecycle 类，保存Lifecycle owners的实际状态，并在生命周期变化时触发事件
  • Lifecycle observers，观察生命周期状态并在改变时执行任务
  • Lifecycle observersimplement LifecycleObserver 接口.

Configuration changes

当用户设备状态发生巨大改变时，就会发生配置变更(Configuration changes),系统最简单的方法完全关闭并重建

配置变更例子：

  • 语言更改，布局要适应不同的文本
  • 设备方向变化，布局要适应新的方向
  • 外设扩展


Architecture components

App architecture

App 架构 是设计应用程序类的一种方式

  • UI controller
    • 基于UI的类，如Activity
    • 应仅包含处理UI和操作系统交互的逻辑
    • Don’t put decision-making logic into the UI controller.
  • ViewModel
    • 保存与ViewModel关联的Activity或Fragment中显示的数据
    • Performs the decision-making.
  • ViewModelFactory
    • ViewModelFactory实例化ViewModel对象

ViewModel

旨在存储和管理与UI相关的数据，每个ViewModel都与一个Fragment或者Activity相关联

ViewModel关联 UI controller：

始终使用ViewModelProvider创建ViewModel对象，而不是直接实例化ViewModel的实例

直接实例化ViewModel，会在配置变更时，反复创建

ViewModelProvider 工作流程：

  • 如果ViewModel存在，则ViewModelProvider返回一个现有的ViewModel；反之创建一个新的ViewModel
  • 只创建与给定范围(Activity或Fragment)关联的ViewModel实例
  • 给定范围存活到什么时候，ViewModel就存活到什么时候

保存配置变更数据:

  1. 将关联范围的UI数据保存到ViewModel中
  2. 关联范围的方法体中，处理数据的代码迁移到ViewModel中，更新UI的代码则保留
  3. ViewModel不应包含对Activity，Fragment或View的引用，因为其无法在配置更改后保留下来

LiveData

LiveData是可观察到的数据持有者类，具有生命周期意识

  • LiveData是可观察的，这意味着当LiveData对象保存的数据发生更改时会通知观察者。
  • LiveData持有数据，LiveData是可以与任何数据一起使用的包装器
  • LiveData具有生命周期感知功能，这意味着它仅更新处于活动生命周期状态（例如STARTED或RESUMED）的观察者

MutableLiveData是一个LiveData，其值可以更改。MutableLiveData是一个通用类，可以指定其保存的数据类型。
